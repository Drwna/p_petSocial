# 宠物社交平台（PetSocial）常见答辩问题清单

> 按照"技术实现 / 架构设计 / 业务理解 / 项目工程 / 创新与不足"五大类整理。
> 每题包含 **问题** + **建议回答要点**，可结合自己的实际情况调整措辞。

---

## 一、技术实现类

### Q1：为什么选用 Express 而不是 Koa / NestJS？

**要点**：
- Express 生态最成熟，中间件丰富，学习成本低，适合快速迭代。
- Koa 更现代但需要自己组装中间件；NestJS 偏重后端工程化，对本项目体量过重。
- Express 5 已支持原生 async/await，能满足异步控制需求。

---

### Q2：JWT 是怎么做鉴权的？过期了怎么办？

**要点**：
- 登录成功后服务端用 `jsonwebtoken` 签发 Token，包含 `userId` 等 payload，默认 7 天过期。
- 客户端在 `Authorization: Bearer <token>` 头中携带。
- 中间件 `auth.js` 解码 Token 并挂到 `req.user` 上；解码失败返回 401。
- 前端 `request.js` 收到 401 后清空本地存储并跳转登录页。
- **拓展**：可引入 refresh token 机制延长会话；当前为简化处理。

---

### Q3：邮箱验证码是怎么实现的？有没有防刷？

**要点**：
- 使用 `nodemailer` + QQ SMTP 发送验证码。
- 验证码及过期时间存于 `VerificationCode` 表。
- 防刷措施可拓展：同一邮箱 60 秒内仅能请求一次、单 IP 每日上限、加入图形验证码等（可作为"未来工作"提及）。

---

### Q4：Sequelize 的关联是怎么定义的？为什么要分多对一、多对多？

**要点**：
- 在 `models/index.js` 集中定义所有关联，避免循环引用。
- 一对一（Account ↔ Pet）：用户和宠物身份强绑定。
- 多对一（Post → Pet）：一只宠物发多篇帖子。
- 多对多（Post ↔ Topic）：一篇帖子可带多个话题，通过中间表 `PostTopic`。
- 自连接（Pet ↔ Pet）：关注 / 拉黑使用 `Follow`、`BlockPet` 表。

---

### Q5：图片上传怎么实现？有没有大小或格式限制？

**要点**：
- 使用 `multer` 解析 multipart 表单。
- 文件保存在 `server/src/public/images/`，通过 `express.static` 暴露。
- 可配置 `limits`（fileSize）和 `fileFilter`（mimetype）做限制。
- **生产改进**：改用对象存储（OSS / S3）+ CDN 减轻服务器压力。

---

### Q6：UniApp 怎么做到一套代码多端运行？

**要点**：
- UniApp 把 Vue 3 模板编译成各端对应的产物：H5 用 Web 标准、微信小程序用 WXML+WXSS、App 端用 weex。
- 通过 `npm run dev:h5 / dev:mp-weixin / dev:app` 选择目标平台。
- 平台差异通过条件编译 `#ifdef MP-WEIXIN` 处理。
- API 调用使用统一的 `uni.*` 命名空间，由框架做平台适配。

---

### Q7：前后端是怎么通信的？有没有处理跨域？

**要点**：
- 前端使用 `uni.request` 发起 HTTP 请求，封装在 `frontend/src/utils/request.js`。
- 后端通过 `cors` 中间件全局允许跨域。
- 后台 admin 使用 Vite proxy `/api` → `http://localhost:3001`，开发期无需 CORS。
- 生产环境通过 Nginx 同源转发即可避免 CORS。

---

### Q8：为什么 dataDB 用 MongoDB 而不是 MySQL？

**要点**：
- 宠物百科条目结构灵活、字段不固定，文档型数据库更合适。
- MySQL 关系建模会拆出大量空字段或扩展表，维护成本高。
- 业务上百科与社交逻辑独立，拆成微服务有利于隔离和单独伸缩。

---

## 二、架构设计类

### Q9：为什么把项目拆成 4 个模块？

**要点**：
- 职责分离：API、移动端、后台、百科互相独立。
- 技术异构：用 MySQL 还是 MongoDB 可以分别选择。
- 部署灵活：每个模块可独立部署、独立扩容。
- 团队协作：模块清晰便于多人并行开发。

---

### Q10：admin 后台为什么不直接合并到 server？

**要点**：
- 前后端分离已是行业主流，便于独立迭代界面。
- admin 使用 Vue 3 + Element Plus 构建 SPA，server 专注 API。
- 避免在 Express 中嵌入大量模板渲染代码，保持后端纯净。

---

### Q11：高并发下系统会不会有性能瓶颈？

**要点**：
- 当前主要瓶颈在 MySQL 单实例和 Node 单进程。
- **优化方向**：
  1. 引入 Redis 缓存热点帖、Session、积分余额。
  2. Node 多进程（PM2 cluster）+ 负载均衡。
  3. 数据库读写分离、添加索引（如 Post.createdAt、Follow.followerPetId）。
  4. 静态资源 CDN，图片走对象存储。
- 当前为本科毕设规模，未做性能压测，可作为未来工作。

---

### Q12：如何保证数据一致性，比如积分兑换？

**要点**：
- 兑换流程涉及"扣积分 + 生成兑换记录 + 商家库存减一"。
- 使用 Sequelize 的 `sequelize.transaction()` 包裹三步操作，任一失败回滚。
- 写入 `PointLog` 流水保证可追溯，便于对账。

---

### Q13：关系图谱很复杂（关注 / 拉黑 / 不感兴趣），怎么避免互相影响？

**要点**：
- 各机制写入独立表（Follow / BlockPet / PostDislike），互不污染主表。
- 信息流查询时，先取关注集合，再排除拉黑、踩、不感兴趣的帖子。
- 可在 SQL 层用 `NOT IN` 或左连接过滤；后续可用 Redis Set 提速。

---

## 三、业务理解类

### Q14：你这个项目和小红书 / 微博的差异化在哪里？

**要点**：
- **身份层面**：以"宠物"为社交主体，而非以"人"。
- **业务闭环**：嵌入了商家、礼品兑换、线下活动这套"线上分享 → 线下消费"的链路。
- **垂直场景**：分类、话题、百科都围绕宠物，非通用社交。

---

### Q15：积分如何获得？规则是什么？

**要点**：
- 当前实现：发帖、点赞、被关注等行为可累积积分（具体写在 PointLog 业务规则中）。
- 后续可拓展：每日签到、活动签到、邀请好友。
- 所有积分变动写入 PointLog 便于审计与运营调优。

---

### Q16：商家审核怎么做的？会不会有恶意商家？

**要点**：
- 商家提交申请后状态为"待审核"，由管理员在后台手动审核通过。
- 审核内容包括营业执照、店铺名称、联系方式（字段可在 Merchant 模型中扩展）。
- 通过后才有发布礼品 / 活动的权限。
- 管理员可随时下架违规礼品或撤销商家资格。

---

### Q17：内容审核怎么做？会不会有违规帖子？

**要点**：
- 当前为"事后审核"：用户举报 + 管理员后台删除。
- **未来可改进**：接入第三方文本 / 图片审核 API（如阿里云内容安全），实现自动化先审后发。
- 同时利用"踩"、"不感兴趣"做用户侧自治。

---

## 四、项目工程类

### Q18：项目代码量多少？开发周期多久？

**要点**（按实际填写）：
- 后端约 X 行，前端约 Y 行，后台约 Z 行；总计约 N 万行。
- 开发周期约 N 个月，包含需求调研、设计、编码、测试、部署。
- 单人完成所有模块。

---

### Q19：你遇到过什么印象深的 bug？怎么解决的？

**要点（可选其一）**：
- **积分未实时更新**：前端从缓存读取旧值，导致兑换后界面不刷新。修复方法：兑换接口返回最新余额，前端用响应数据替换本地状态（commit `8f0e806`）。
- **跨端兼容**：UniApp 在小程序端不支持某些 H5 API，通过条件编译解决。
- **Sequelize 关联多对多**：一开始忘了在 `belongsToMany` 中显式声明 `through` 表，导致中间表字段名不一致，调试半小时才找到。

---

### Q20：项目用到了哪些版本控制 / 协作工具？

**要点**：
- Git 管理代码，commit 信息按 `feat / fix / style / refactor / docs` 分类。
- 远程仓库（GitHub / Gitee）保存历史。
- 配合 Issue 或 TODO 管理任务进度。

---

### Q21：怎么测试的？有没有写单元测试？

**要点**：
- 接口层使用 **Postman** 手动测试，覆盖关键路径。
- 前端使用浏览器 + 微信开发者工具多端调试。
- 当前无自动化单元测试，是项目的不足之一。
- **未来工作**：使用 Jest 编写关键业务的单元测试，使用 Supertest 做接口集成测试。

---

### Q22：如何部署上线？

**要点**：
- 后端 Node 服务用 `pm2` 守护进程。
- Nginx 做反向代理 + HTTPS 证书。
- MySQL / MongoDB 部署在云服务器或 RDS。
- 前端 H5 打包后部署到对象存储 + CDN。
- 后台 admin 同样静态资源部署，通过 Nginx 转发 `/api` 到后端。

## 五、创新与不足类

### Q23：本项目有什么创新点？

**要点**：
1. 以"宠物"为社交身份的设计思路。
2. 多端 + 多库 + 多服务的现代 Web 架构实践。
3. 社区 + 商家 + 活动 + 积分四位一体的业务闭环。
4. 多维内容过滤机制（踩 / 不感兴趣 / 拉黑），改善信息流体验。

---

### Q24：项目有哪些不足？怎么改进？

**要点**：
- 缺少自动化测试 → 引入 Jest + Supertest。
- 推荐算法简单（按时间倒序）→ 基于行为日志做协同过滤或召回排序。
- 没有 IM 即时通讯 → 接入 WebSocket / Socket.IO 实现私信。
- 单实例部署，未做高可用 → 引入 PM2 cluster + 负载均衡 + 数据库读写分离。
- 缺少监控告警 → 引入 Prometheus + Grafana 做指标采集。
- 缺少 Redis 缓存 → 加入热点数据缓存层。

---

### Q25：未来如何商业化？

**要点**：
- 商家入驻费 / 抽佣 / 广告位。
- 高级会员订阅（专属头像、扩容相册、专属话题）。
- 与本地宠物医院、宠物用品店合作做团购 / 优惠券。
- 数据增值（聚合用户偏好后的市场分析报告，注意隐私合规）。

---

### Q26：相比毕设的常见社区项目，你的难点在哪里？

**要点**：
- 多端代码同时维护（H5、小程序、App），需处理平台差异。
- 双数据库整合（MySQL + MongoDB）以及微服务边界划分。
- 业务复杂度高：不只是 CRUD，还包含积分流水、商家审核、活动报名等真实业务流程。
- 数据模型耦合度高：自连接 + 中间表 + 多角色权限。

## 六、可能的"刁钻问题"应对

### Q27：你的数据库初始化用 `sync({ alter: true })`，生产可以用吗？

**要点**：
- `alter: true` 在生产是危险的，可能造成数据丢失。
- 生产应使用迁移脚本（`sequelize-cli` 的 migrations），版本化管理 schema。
- 当前仅用于本地开发与演示阶段的便捷初始化。

---

### Q28：JWT Token 泄露怎么办？

**要点**：
- Token 一旦泄露在过期前都有效，是无状态鉴权的固有风险。
- **缓解方案**：HTTPS 传输；Token 短期 + Refresh Token；维护"黑名单"撤销列表（牺牲一些无状态性）；敏感操作二次验证。

---

### Q29：项目中有没有 SQL 注入 / XSS 风险？

**要点**：
- 使用 ORM（Sequelize）参数化查询，规避 SQL 注入。
- 用户输入在前端展示前应做转义；当前部分富文本场景可用 DOMPurify 之类做过滤。
- 上传文件需限制 mimetype 与大小，防止脚本上传。

---

### Q30：前端组件复用怎么做的？

**要点**：
- 公共组件放在 `frontend/src/components/`，通过全局或局部注册复用。
- 网络请求统一封装，业务页面不直接调用 `uni.request`。
- 全局样式 / 主题色用 SCSS 变量统一管理。

---

## 七、回答技巧

1. **先承认范围，再展望**："这一点我目前的实现是 X，但生产环境会用 Y 改进"——既体现实事求是，又显示有更深的认知。
2. **用术语 + 用例**：先讲专业名词，再举一个本项目里的具体例子，避免空泛。
3. **结构化作答**：用"先 / 然后 / 最后"或"一是 / 二是 / 三是"组织回答，听感清晰。
4. **不会就直说**：评委问到不熟悉的内容时，承认并说明你的思路（"这部分目前没实现，但我会从 ... 着手研究"），不要硬答。
5. **控制时长**：单题回答 30~90 秒为宜，避免占用过多答辩时间。
